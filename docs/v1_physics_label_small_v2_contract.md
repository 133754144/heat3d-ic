# Heat3D v1 Physics-Label Small v2 Contract

## Purpose

`v1_multilayer_bc_eq_physics_label_small_v2` is a small physics-label smoke subset for the Heat3D v1 physics-label pipeline. It is intended to check whether the reference solver v2 path can generate temperature labels with explicit solver metadata and whether label diagnostics can read that metadata.

This subset is a research reference / smoke diagnostic artifact. It is not a formal benchmark, not model-performance evidence, not OOD generalization evidence, and not an industrial 3D IC thermal simulation dataset.

## Current Scope

The current v2 subset covers the full 16-sample supervised-small manifest:

- train: 10 samples
- valid: 3 samples
- test_smoke: 1 sample
- test_ood_bc: 1 smoke candidate
- test_ood_stack: 1 smoke candidate

The `test_ood_bc` and `test_ood_stack` entries remain diagnostic smoke candidates only. They are not OOD-generalization evidence. The source subset remains unchanged. The v2 subset is written under ignored `data/` paths and must not be committed to Git.

## Label Generation

Each v2 sample is generated by:

1. Copying existing metadata arrays and `sample_meta.json` from `v1_multilayer_bc_eq_supervised_small`.
2. Removing any prior `temperature.npy`.
3. Solving with `rigno/heat3d_v1_reference_solver_v2.py`.
4. Writing a new `temperature.npy`.
5. Writing `label_meta.json`.

The old `v1_multilayer_bc_eq_supervised_small` labels are not overwritten.

## Required Files

Each sample must contain:

- `coords.npy`
- `layer_id.npy`
- `region_id.npy`
- `material_id.npy`
- `k_field.npy`
- `q_field.npy`
- `sample_meta.json`
- `temperature.npy`
- `label_meta.json`

## Required Solver Metadata

`label_meta.json` must include at least:

- `solver_name`
- `solver_version`
- `discretization_type`
- `supported_k_mode`
- `convergence_flag`
- `residual_norm`
- `bottom_dirichlet_error`
- `top_robin_status`
- `side_adiabatic_status`
- `interface_status`
- `energy_balance_status`
- `warnings`

These fields are diagnostics for the research reference path. They are not high-fidelity physics validation results.

## Current Solver Scope

The current solver v2 smoke path supports:

- steady 3D heat conduction
- regular multilayer rectangular stacks
- isotropic `(N,1)` conductivity
- diagonal anisotropic `(N,3)` conductivity
- volumetric heat generation
- top Robin boundary condition
- bottom Dirichlet boundary condition
- side adiabatic boundary condition
- perfect-contact interfaces
- conservative finite-difference / finite-volume-style face treatment
- harmonic-mean face conductivity at k jumps

It does not support explicit TSV / BEOL / bump structures, irregular footprints, contact resistance, transient simulation, multiphysics, or `(N,6)` full tensor conductivity.

## Diagnostics

The label diagnostics smoke may check:

- array shape, dtype, and finite values
- `temperature.npy` shape and finite values
- basic temperature and DeltaT statistics
- bottom Dirichlet simple consistency
- optional `label_meta.json` fields, convergence flag, residual norm, and bottom error

PDE residual, top Robin flux violation, side adiabatic flux violation, interface flux mismatch, and global energy balance remain `requires_numerical_operator` or `not_computed` unless a credible discrete operator is implemented.

## Next Step

After this full-manifest smoke path is stable, the next planned step is to reuse the existing zero-delta, normalized DeltaT, train-valid, and validation-metrics smoke checks with the v2 labels.
